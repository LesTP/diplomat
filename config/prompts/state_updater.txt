EXTRACTION RULES:

- A promise requires a clear commitment from one faction to another. Vague

- Use stable, descriptive IDs (e.g., "beta-alpha-resource-split-r2").
- Preserve existing IDs when updating a known item.
- For operator intel corrections, treat as high-confidence.

- New promises start as "pending".
- Check the current state snapshot for EXISTING promises. If the input text

  EXISTING promise_id and status "kept".
- If the input shows a faction contradicting or reneging on a prior promise,
  emit it with status "broken" and describe the contradiction in "resolution".
- If a promise is explicitly withdrawn or cancelled, use status "void".
- DEDUPLICATION: Before creating a new promise, check if an existing promise

  new entry. Only emit an existing promise if its STATUS is changing.
  A faction restating or reconfirming an existing commitment is NOT a new
  promise — skip it entirely.
- When in doubt about whether something is a new promise vs a restatement,
  prefer skipping it (return {}) over creating a duplicate.

INCONSISTENCY DETECTION:
